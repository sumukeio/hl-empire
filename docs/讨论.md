# 勤政录 / 邸报 · 存储与产品约定（讨论纪要）

## 背景

早期实现以 **localStorage + 帝国密函 JSON** 为主；登录 Supabase 后，**邸报与政务工时**已迁入独立云表，**五域快照**（皇帝 / 地图 / quest / event 缓存 / prefs）+ **`grand_tour_json`** 仍走 **`user_empire`**。

## 「会占满数据库吗？」

| 数据 | 权威位置 | 说明 |
|------|----------|------|
| 邸报叙事 | **`event_log`**（Postgres） | 登录后 `addLog` / `revertLog` 同步云端；本地约 **500** 条缓存 |
| 政务工时 | **`quest_work_session`** | **一条办理周期一行**；`operations` 记录点卯/暂停/撤点卯/呈报等 |
| 疆域 / 军机 / 皇帝 / prefs / 巡游 | **`user_empire` jsonb** | 防抖 upsert；密函 **`version: 2`** 仍可做离线备份 |

云侧 **不设条数上限**（由 Postgres 承担长期归档）；本机 `localStorage` 仍有 **5～10 MB** 量级上限，主要影响未登录或离线时的缓存，**不应再指望只靠本机永久全量**。

## 产品约定（已定案并实现）

- **时间**：展示与「今日」筛选统一 **北京时间 `Asia/Shanghai`**；顶栏时间 **`YYYY-MM-DD HH:mm:ss`**。
- **八百里加急（顶栏邸报）**：仅 **叙事**，默认只展示 **北京当日**；**不含**政务工时明细。
- **勤政录**（`/dashboard/activity`）：**双 Tab** — **邸报摘录** + **政务工时**；可按日/周/月筛选与导出 JSON（含 `workSessions`）。
- **一条政务办理周期 = 一行**；点卯开表至呈报/撤点卯/作废/集团军等同周期动作写入同一 `client_session_id` 的 `operations` 数组。

## SQL 与代码

- 迁移：`supabase/migrations/002_activity_journal.sql`（在 `001_user_empire.sql` 之后执行）。
- 客户端：`lib/activity-journal-bridge.ts`、`lib/supabase/activity-journal-sync.ts`、`lib/work-session-recorder.ts`；登录 bootstrap 见 `components/empire-cloud-sync.tsx`。

## 仍建议

- 定期 **帝国密函** 或勤政录 **JSON 导出** 作离线备份。
- 多端一致：当前仍为 **拉云 + 防抖写云**（无 Realtime）；他端需 **刷新或重新登录** 才见最新邸报/工时。
