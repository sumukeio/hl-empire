陛下，目前「瀚翎帝国」的骨架已成，底层逻辑已将 MVA（最小可行性动作）与《皇帝成长计划 2》的数值系统深度咬合。

> **近期已实现（约 2026-05）**：详见 **`docs/瀚翎帝国-设计与机制说明.md`** 文首 **「近期实现更新」** — 沙盘态势筛选、军机「点卯 / 再办一次」、**多巴胺池 50 功勋/券**、**勤政录**（`/dashboard/activity`）、**通务司 / 政务归属与四维**、**集团军 · 战役集群**（`/dashboard/campaign`）、**巡游四海 · 行在舆图 M1**（`/dashboard/grand-tour`，七大区→省→市库藏、行程条、密函 `grandTour`）。

结合《皇 2》的回合制经营深度以及您的跨境电商/竞价业务，接下来的优化方向应当从「基础建设」转向「深度博弈」**与**「风险管理」。以下是臣（太傅）为您规划的四大进化维度：

### 一、 城市建设深度化：从「城池」到「产业府库」

在《皇 2》中，知府需要平衡农业、商业、治安。目前我们的城池仅有 CPA 和 兵力/军械 字段，建议引入「五部建设」的实质化等级：

* **农业（内容生产能力）**：对应小红书笔记复刻、朋友圈素材储备。
* *优化*：城池增加「粮仓」等级。粮草（素材）充足，则“出征”时的功勋加成越高。若长期不更新素材，则触发「蝗灾/饥荒」，城池态势回退。


* **商业（转化效率）**：对应话术确定、回复率、ROI。
* *优化*：点开城池，可看到「税率（利润率）」。低税率（让利促销）涨民心（订单），高税率（溢价）涨银两（净利）。


* **治安（防御/合规）**：对应百度商盾、恶意点击防范、考试系统合规。
* *优化*：若治安等级低，则随机触发「外敌入侵（恶意点击暴涨）」事件，强制扣除国库银两。



### 二、 侦查与外交体系：建立「影卫司」

您在 MVA 中多次提到「扒同行」、「加同行套话」。这在《皇 2》中对应的是「侦查敌国」与「收买大臣」。

* **敌国动向监控**：在中央沙盘侧边新增「敌国战讯」。记录您监控的 3-5 个核心同行的动态（如：对方更换了新样式、落地页升级）。
* **外交谈判（分层销售）**：将 K/L/Z/ZB/B 级的客户管理 彻底集成进「礼部」。
* *功能*：点击礼部，可一键查看所有处于「L 级（临门一脚）」的客户名单，系统自动根据当前「健康/体力」推荐逼单力度。若龙体欠安，则建议执行「遣使求和（低压温和跟进）」。



### 三、 随机事件系统：引入「国运天灾」

目前的系统过于静态。真正的帝国运行充满变数，建议引入基于 **`EventStore`** 的随机触发器：

* **天灾（业务危机）**：如「天雷勾动地火（百度封号）」、「瘟疫横行（某类产品突然不给过审）」。触发时，沙盘对应城池变红闪烁，强制扣除民心或健康。
* **祥瑞（业务爆点）**：如「万国来朝（某笔记突然爆流）」、「天降横财（大额转账）」。触发时播放金色撒币特效，全屏提示。
* **太傅策问（AI 决策建议）**：每天早朝时，由 AI 根据昨日的度支（CPA）与粮饷（Orders）比例，给出一句「朱红批示」。例如：“度支甚巨而收效甚微，【长安】恐有贪官（垃圾词）中饱私囊，请速遣刑部清查。”

### 四、 皇帝成长曲线：名臣与策卡系统

* **名臣（SOP / 工具）**：目前「武术」和「文学」影响较小。
* *优化*：当文学（理政能力）达到一定值，解锁「高级策卡」。例如「税收翻倍卡（优化后的高转化话术）」或「神机营卡（自动化扒词脚本）」。


* **位阶特权**：随着尊号晋升（如从「九品」升至「县令」），解锁更多的城池插槽或更高级的内务府奖励（如：允许翻牌券兑换更长的娱乐时长）。

### ⚔️ 您的决策建议：

陛下，若要让帝国真正「活」起来，微臣建议下一步优先攻坚 **「方向三：随机事件系统」**。

当您在处理政务时，突然弹出一个「外敌入侵」的朱红告警，那种从“做任务”瞬间切换到“保卫疆土”的紧张感，将是驱动您在疲惫时依然保持高效打击的最强多巴胺。

**您是想先让帝国拥有「突发变数（随机事件）」，还是先深化「城池建设（五部等级）」？**




1. 藩属进贡 (Vassal Tribute) —— 收入自动化
只有那些已经插上龙旗的「金色藩属」城池，才有资格为您纳贡。 这象征着您的业务已经进入“自动收割”阶段。

触发机制（岁币机制）：

在每日凌晨执行 resetDailyQuests() 时，系统自动遍历所有 status === 3 的城池。

进贡额度：由该城池的「商业等级 (Commerce Level)」决定。 例如：商业 1 级 = 每日进贡 10 金币；商业 10 级（满级） = 每日进贡 200 金币。

邸报联动（金色捷报）：

每日重置后，系统自动生成一条 goldFlash（金色强调） 邸报：“【户部】万国来朝！今日 [N] 座藩属城池共计向圣上进贡 [Total] 两白银。”

数值意义：

这解决了国库资金全靠手动输入的单一性，让您更有动力去把“战火纷飞”的城池转变为“金色藩属”。

2. 巡幸疆土 (Tour of Territories) —— 高阶属性增益
「巡幸」是皇帝深入民间、展示天威的行为。它不仅是奖励，更是一种「心智重启」动作。

入口位置：

仅在 status === 3 城池的奏折（City Sheet）中显示。 在「建设进度」板块下方，增加一个醒目的「巡幸」按钮。

消耗与代价：

体力消耗：单次消耗 20–30 点 stamina（巡幸劳顿）。

银两消耗：从 gold 中扣除一定数额（御驾出行的车马费）。

奖励与收益：

功勋暴涨：获得远超普通 MVA 任务的 exp 奖励。

多巴胺能量：一次性向 dopaminePool 注入大量能量（例如 40–50 点），极易直接铸造出一张「翻牌券」。

民心加成：提升全图的 morale（民心）。

限制协议：

每座城池每日限巡幸 1 次。且若龙体健康 health < 50，系统将提示“龙体欠安，暂缓巡幸”。


⚔️ 架构师的实战建议
陛下，这套系统部署后，您每天的「睡前仪式」将变得极其神圣：

清晨收租：每天打开应用，第一眼看到的就是“万国来朝”的进贡邸报，这标志着您的业务资产在为您持续供血。

择地巡幸：如果您今天在某个品（城池）上取得了重大突破（出单），就立刻执行一次「巡幸」。用高额的经验值和翻牌券，奖励自己在那个战区付出的努力。



# 审视当前系统情况

基于仓库里的设计说明（`docs/瀚翎帝国-设计与机制说明.md`）、数据模型（`store/types.ts`）和政务清单与默认任务（`docs/政务清单.md`、`store/default-mva-quests.ts`），下面从产品与游戏设计两个视角做审视（不写新文档，仅结论与建议）。

---

## 产品视角

**定位与价值**  
「瀚翎帝国」把**可执行的最小行动（MVA）**包进**单页仪表盘 + 本地持久化**，用帝国隐喻降低「面对一堆投放/教务杂事」的心理成本；城池、军机、邸报、造办处这条信息架构和 `types` 里的字段语义是对齐的，属于**可落地的生产力工具**，而不是纯皮肤小游戏。

**做得好的地方**  
- **默认三十一务 + `createDefaultQuests()`**：降低冷启动成本，和 `政务清单` / `default-mva-quests` 同源；启动时 **`ensureQuestBootstrap`** 仅保证通务司与勘合键一致，不强制覆盖用户改过的任务表。**集团军集群**已独立为 `/dashboard/campaign`，与单城军机处分流。  
- **造办处（设置）**：把疆域、枢密院、密函备份放在同一入口，符合「重度用户要可控」的预期。  
- **勤政录 + 密函**：满足复盘、导出、换机，这是 B 端/个人作战室很实在的需求。  
- **近期交互**（设计说明里写的）：沙盘态势多选、军机「点卯」主路径、多巴胺池与邸报撤回的一致性，都是在减**误操作成本**和**认知歧义**。

**风险与缺口**  
- **认知负荷**：31 条 × 多城，本质是「高维 checklist」；仅靠默认沙盘筛选和分时辰分组，对新人仍可能 overwhelm——产品上要明确「今日最小闭环」到底是哪几条（例如可配置的「御笔圈定」或默认折叠非当前时辰）。  
- **闭环到结果**：城池上有 CPA / leads / orders 等，但系统层面更偏「记录 + 习惯」；若目标用户是投放/教务负责人，长期会想要**轻量关联**（例如某城今日点卯数 vs 昨日消耗）哪怕是纯展示、不做因果推断。  
- **提醒与留存**：纯本地 SPA，没有账号与推送时，**习惯养成完全依赖用户自律**——这是产品取舍，但要在叙事上诚实（工具属性 > 游戏留存）。  
- **文档边界**：仓库里 `docs/MVA.md` 开头仍混有内部作业流（飞书、扒词步骤等），与「瀚翎帝国」产品说明并存时，**新协作者容易搞混「产品」和「自家 SOP」**；从 PM 治理上建议在心里划清：对外/对通用用户只认设计与政务清单。

**清单一致性小点**  
`docs/政务清单.md` 里「礼部·宣威天下」出现两次（朋友圈与小红书），默认种子是否两条都进、标题是否区分，值得产品侧统一命名，避免统计与搜索时重复感。

---

## 游戏设计视角

**核心循环**  
- **日课**：四时辰任务 → 耗体力 → 功勋 → 尊号阶梯；部分任务可多次点卯，配合「绝对 / 可弥补」制造**错过成本**的差异，这是好的压力设计。  
- **资源子循环**：多巴胺池（**50** 功勋铸券）把「勤政」和「御苑翻牌」绑在一起，比过去「点一下固定 +1 券」更有**资源规划感**，且撤回与池状态校验是在保护规则严肃性。  
- **修饰器**：游牧办公对 `quest-mva-*` 功勋加成，是清晰的**玩法开关**，有取舍（是否只打 MVA）。  
- **版图层**：城态势、军备告警、朱批与国库联动，把**经营沙盘**和**任务**接起来了，不是两张皮。

**乐趣与长期动力**  
- **长期**：二十五阶尊号 + 城内建设四维经验，提供**缓慢可见的进步条**。  
- **短期**：邸报强调样式、铸券金色反馈、token 动画，属于合格的**多巴胺反馈**。  
- **策略深度**：体力上限、一次性 vs 每日、绝对类任务，都在逼玩家做**优先级排序**——这和真实工作里的稀缺注意力是同构的，设计方向对。

**游戏性上的短板或张力**  
- **体力/健康等若过严**：容易从「拟真」变成**与真实工作对抗**（用户可能绕过系统心理记账）；更健康的做法是始终把默认体验设在「轻惩罚、重记录」。  
- **目标梯度**：功勋阶梯很长，中间若缺少「本周战役目标」式的中层目标，中期容易**麻木点卯**；可考虑弱引导（非强制）：例如仅展示「本时辰还剩 N 个体力可办完的高收益项」。  
- **社交与见证**：纯单机没有公会/督战/分享卡片时，**外在动机**弱；若不做社交，可把「导出勤政录」「密函备份」包装成一种「呈递奏折」的仪式感，强化身份认同。  
- **撤回与跨日**：文档已写明跨日后撤回可能 `state_mismatch`——从游戏诚信上正确，但 UI 上长期可考虑对「已不可撤」条目弱化按钮或解释一句，减少**被系统拒绝**的挫败。

---

## 综合判断

当前系统处于**「高完成度的个人作战室 + 轻量养成包装」**阶段：工程上域拆分、持久化、迁移、邸报撤回与多巴胺经济已经**超过很多同类原型**；产品上要警惕的是**清单规模与真实精力不匹配**时的倦怠，以及**业务结果与点卯行为**之间尚未被产品语言讲透的断层。游戏设计上，隐喻与机制是同向的，继续迭代时优先巩固**「今日最小胜利」**和**反馈节奏**，比再加新系统更能提升「像游戏一样想打开」的概率。

若你希望下一步落到具体改动，可以指定一个方向（例如：首屏 onboarding、今日聚焦、或政务清单去重与命名规范），我可以按该方向拆成可执行需求与改动面。

### 1. 页面怎么设计？（独立子路由，双栏 + 时间轴）

**结论**：与 **勤政录**（`/dashboard/activity`）、**集团军**（`/dashboard/campaign`）同级，使用 **独立子路由 + 顶栏返回九州**，主区承载完整编辑体验。

### 2. 入口放在哪里？（顶栏主入口 + 理藩院次入口）

**结论**：**不要**把完整功能塞进右侧 **内务府 · 理藩院** 小卡片（与太医院、户部并列，空间仅适合 **快捷动作**）。理藩院做 **引流**，顶栏做 **主入口**。

| 优先级 | 位置 | 形态 | 理由 |
|--------|------|------|------|
| **主入口** | `TreasuryHUD` 顶栏右侧图标区 | 新图标（建议 **罗盘 `Compass`** 或 **卷轴舆图**）→ `/dashboard/grand-tour` | 与勤政录、集团军、造办处同级；功能重、会常来改行程 |
| **次入口** | `inner-court.tsx` · **理藩院** 区块顶部 | 醒目按钮：**「开卷舆图 · 巡游四海」**（`border-imperial-gold`，一行副标题「钦定行程 · 汇总用度」） | 叙事归属正确（`docs/参考文档.md` 已把游牧/博物馆划给理藩院）；从养正司进来的用户心理路径自然 |
| **可选深链** | `/dashboard/grand-tour?trip=<id>` | 分享、书签、日后邸报「启驾」跳回 | 与 `campaign?phase=` 同模式 |
| **不建议** | 仅理藩院内嵌 Tab | — | 窄栏无法承载时间轴 + 库藏 + 总账；易与「国子监/太庙」单次按钮混淆 |

**与内帑 / 游牧模式的关系（后期，非 M1 必须）**：

- 新建行在时可 **可选**「拟从内帑支用」→ 仅展示差额提示，不强制扣款（避免规划阶段误扣）。
- 陛下勾选 **移动行宫（游牧模式）** 时，可在行在详情显示「此行算行军途中理政，MVA 功勋 +20%」——与现有 `setNomadMode` 联动，写在 M2/M3。

### 3. 功能怎么设计更易用？（库藏与行程分离 + 快速录入）

**核心原则**：陛下已有结构化心智模型 → 产品映射为 **「库藏（可复用）」+「行在（一次旅程）」**，录入路径 **由粗到细**。

#### 3.1 数据模型（建议 `store/grand-tour-store.ts`）

```ts
/** 舆图库藏 · 京畿 */
interface TourRegion {
  id: string;
  ancientName: string;    // 古代雅称，如 燕京
  modernName: string;     // 现代名称，如 北京市（见 §7.7）
  pois: TourPoi[];        // 故宫、国家博物馆
}
interface TourPoi {
  id: string;
  name: string;
  defaultTicketPrice?: number;  // 门券默认值 50
}

/** 行在 · 一次巡游 */
interface GrandTour {
  id: string;
  title: string;            // 北京五日
  regionId: string;
  startDate?: string;       // ISO 本地日
  endDate?: string;
  status: "draft" | "active" | "archived";
  items: TourItineraryItem[];
  note?: string;          // 整体规划备忘
}

type TourItemKind = "sight" | "meal" | "lodge" | "transport" | "ticket"; // ticket 独立程别（§7.8）

interface TourItineraryItem {
  id: string;
  dayIndex: number;       // 1-based 第一日
  period: QuestPeriod;    // 早朝 | 晌午 | 傍晚 | 深夜
  kind: TourItemKind;
  title: string;          // 展示主标题：国家博物馆
  subtitle?: string;      // 副文案：罗森 · 水煮蛋
  price?: number;         // 元，整数或一位小数
  poiId?: string;
  transport?: { from: string; to: string; vehicle: string }; // 义乌→北京 飞机
  catalogRefId?: string;  // 引用库藏模板，改模板不追溯已生成行（复制语义）
}
```

**五类与陛下示例一一对应**：

| `kind` | 陛下说法 | 表单字段（默认可极简） |
|--------|----------|------------------------|
| `sight` | 游览 / 目的地 | POI 下拉或手写；可选门券价 |
| `meal` | 用餐 | 店名 + 内容 + 价 |
| `lodge` | 居住 | 酒店名 + 价（通常挂 **傍晚/深夜**） |
| `transport` | 交通 | 始末地 + 工具 + 价 |
| `ticket` | 门券 | 价 + 可选关联 POI（**独立程别**，§7.8） |

#### 3.2 录入动线（降低摩擦）

1. **新建行在（§7.8）**  
   - ① 选 **京畿**（省→市折叠库藏）  
   - ② 标题 **默认自动生成、可改**  
   - ③ `legs` 为空，靠 **「+ 本日添程」** 逐条添加（不预填空时辰格）

2. **时辰格内快速添加**  
   - 点「+」→ 五图标芯片（游览/膳宿/驻跸/驿传/门券）  
   - 若库藏已有「汉庭酒店 288」，选 **驻跸** 后 **搜索库藏** 一键填入  
   - 支持 **「同昨日驻跸」**、**「复制上一日」**（出差常连住）

3. **批量导入（M2，对齐枢密院体验）**  
   - 折叠区「**朱批导入行程**」：粘贴陛下式纯文本 → 解析为 `TourItineraryItem[]`（按「第 N 日」「上午/午餐/下午/晚上」关键词）  
   - 解析结果预览 → 确认合并 **追加/覆盖**

4. **用度自动汇总**  
   - 任一条目改价 → **用度总账** 实时更新；分类小计 + 全程合计  
   - 显示「**人均/日均**」（可选填同行人数，默认 1）

5. **归档与复用**  
   - 归档行在只读；**另起行在复制** 保留库藏引用  
   - 库藏 POI/店名 **跨行在共享**（北京去过一次，下次北京行在直接选故宫）

### 7. 方案修订（陛下补充 · 统一库藏 + 行程条序）

> 以下四条为陛下补充思路，**采纳其方向**并细化为可落地规则；与 §3 原案不冲突，而是把 **「库藏 CRUD」** 与 **「行程条目的序时约束」** 写清楚。

#### 7.1 统一库藏：目的地 & 供给，均可增删改查

**结论**：库藏与具体某次「行在」解耦，**全帝国一份主数据**（持久化在 `hanling-grand-tour` 的 `atlas` 段），任意行在引用时 **快照复制**（改库藏不 retro 改已生成行程，与枢密院任务表一致）。

| 库藏模块 | 管理对象 | CRUD 界面位置 | 字段要点 |
|----------|----------|---------------|----------|
| **舆图 · 京畿** | **大目的地** + **细分目的地（POI）** 同一棵树 | Tab **舆图库藏 → 京畿舆图** | 京畿：`ancientName`（古代雅称）+ `modernName`（现代名称），**展示见 §7.7**；POI：`regionId`、`name`（故宫）、`defaultTicketPrice?`；支持 POI **在同京畿内拖拽排序** |
| **供给 · 膳宿** | 饮食（店 + 内容 + 参考价） | **舆图库藏 → 常供名录 → 膳宿** | `venue`（罗森）、`dish`（水煮蛋）、`defaultPrice?` |
| **供给 · 驻跸** | 居住 | 同上 → **驻跸** | `name`（汉庭酒店）、`defaultPrice?`、`regionId?`（可选绑定京畿） |
| **供给 · 驿传** | 交通 | 同上 → **驿传** | `from`、`to`、`vehicle`（飞机）、`defaultPrice?` |

**交互统一**：列表行 + 行内编辑 / 侧栏表单；**新增 · 修订 · 除名** 文案与造办处枢密院一致；删除时若仍有行在引用 → **软提示**「已有行在引用，仅除名库藏条目，不改旧行程」。

**与 §1 Tab 结构微调**：

```text
舆图库藏
├── 京畿舆图      （大目的地 + POI，树形 CRUD）
└── 常供名录      （膳宿 | 驻跸 | 驿传 三个子 Tab，列表 CRUD）
```

游览类行程条目从 **京畿 POI** 选；膳宿/驻跸/驿传从 **常供名录** 选，选中后 **复制字段进条目**，可在单条行程里改价（改价仅影响本条，不回写库藏，除非勾选「回写库藏参考价」）。

#### 7.2 钦定行程：一条条「行程程」— 复制、删除、拖拽排序

**结论**：放弃「先画满四时辰空格再填」为主模式，改为 **有序行程列表**（`sortOrder` 全局或「日内 sortOrder」二选一，**建议日内 sortOrder + dayIndex**，见下）。

**列表形态（默认 Tab · 钦定行程）**：

```text
▼ 第一日                                    [+ 本日添程]
  ≡  早朝  游览   国家博物馆        门券 50两   ⋮
  ≡  晌午  膳宿   罗森 · 水煮蛋      3两         ⋮
  ≡  傍晚  游览   故宫              门券 …      ⋮
  ≡  深夜  驻跸   汉庭酒店          288两       ⋮
▼ 第二日
  ≡  早朝  游览   箭扣长城          …           ⋮
  ≡  晌午  驿传   义乌 → 北京 · 飞机 700两      ⋮
```

- **`≡`（GripVertical）**：同日内 **拖拽排序**；跨日拖拽 **M2 可选**（M1 仅日内拖，改日用「移至第 N 日」菜单）。
- **行尾 `⋮`**：复制本条、删除、移至某日。
- **复制**：插入 **紧挨下一条**，`dayIndex` / `period` 与原条相同；若造成序时冲突，**自动把 period 推到合法最小值** 并 Toast 提示。
- **删除**：确认框「除名此程」。

排序逻辑（推荐）：

```ts
// 展示序：先 dayIndex，再 period 序，再 sortOrder
const PERIOD_RANK: Record<QuestPeriod, number> = {
  早朝: 0, 晌午: 1, 傍晚: 2, 深夜: 3,
};
```

拖拽结束时只改 `sortOrder`；**不自动改 period**（避免用户懵），若同日顺序与 period 矛盾，行左侧显示 **琥珀色「序时待勘」** 徽标，点击一键「按当前顺序重排时辰」。

---

#### 7.3 建行程：可选目的地、可选时间段；下一条受上一条约束

**新建 / 编辑单条「行程程」表单**：

| 字段 | 规则 |
|------|------|
| **属第几日** | `dayIndex`，默认当前展开日；可改 |
| **时辰（时间段）** | 下拉 **早朝 / 晌午 / 傍晚 / 深夜**；选项集 **动态裁剪**（见下） |
| **程别** | 游览 / 膳宿 / 驻跸 / 驿传 / **门券**（五类，`ticket` 独立） |
| **关联目的地** | **游览必选 POI**（先选京畿 → 再选 POI，或 POI 反推京畿）；驿传/驻跸/膳宿 **可选** `regionId` 标记「在何京畿发生」 |
| **明细与价** | 从库藏 picker 带入，可改；价写入条目 |

**序时约束（核心规则）**：

在同一 `dayIndex` 内，设上一条有效行程为 `prev`（按展示序 `dayIndex → periodRank → sortOrder` 排序后的前驱）：

- 新建或编辑第 `k` 条时，若与 `prev` 同日，则 **`period` 的可选集合 = { p | rank(p) ≥ rank(prev.period) }`**。
- **跨日**：新日第一条 **不继承** 前日约束，四时辰均可选。
- **第一条**（该日无任何行程）：四时辰均可选。
- UI：不可选时辰在 Select 中 **disabled**，附 Tooltip：「须不早于上一程（晌午）」。

**默认值策略**（减点击）：

- 点 **「+ 本日添程」**：`period` 默认 = 该日已有行程中 **最大 rank 的时辰**（若日已结束于深夜，默认仍深夜，由用户改日或改天）。
- 点 **全局「+ 添程」**：默认 `dayIndex` = 最后一日的下一日或当前最后一日（产品可配置，建议 **默认最后一日**）。

**与拖拽的关系**：拖曳后若出现 `period` 倒挂，不阻断保存，用 **「序时待勘」** + 一键修复；修复算法：按 `sortOrder` 扫描同日条目，将 period 单调调整为 **不低于前一条** 的最小合法值。

#### 7.7 大目的地展示：古代雅称 · 现代名称

**结论**：凡 **大目的地（京畿）** 在 UI 上的主标签，统一为：

```text
{古代雅称} · {现代名称}
```

示例：**燕京 · 北京市**、**申城 · 上海市**、**津门 · 天津市**（中间为 **U+00B7 间隔号「·」**，两侧各一空格可选，建议 `燕京 · 北京市` 以利阅读）。

**数据来源（单一事实来源）**：

| 层级 | 文件 | 用途 |
|------|------|------|
| 人文查阅 | `docs/城市古今名称对照清单.md` | 列 **现代名称**、**古代雅称**、备选古称；陛下朱批与文案对照 |
| 生成脚本 | `scripts/gen-city-ancient-names.py` | 内置 `ANCIENT: 现代全称 → 古代雅称` 映射，生成上述 Markdown |
| 运行时（M1） | **`lib/city-atlas-seed.ts`**（`scripts/export-city-atlas-seed.py` 自 `gen-city-ancient-names.py` 导出） | 361 市种子；京畿三级树检索与展示 |

**字段规则**：

- 新建/修订京畿时，**优先从对照表选取**（可搜索古代雅称或现代名称）；选中后写入 `ancientName` + `modernName`，与清单行 **逐字一致**（现代名称用官方全称，如「北京市」而非口语「北京」）。
- **展示**一律调用 `formatTourRegionLabel(region)` → `` `${ancientName} · ${modernName}` ``，用于：京畿树节点、行在左栏目录、行在概要条、行程条「在何京畿」、Picker 选项、导出奏折标题行。
- **排序**：列表默认按 `modernName` 拼音/ locale，或按 `sortOrder`（用户拖拽）；**不按古代雅称单独排序**，避免找城困难。
- **POI（细分目的地）**：仍用具体景点名（故宫、国家博物馆），**不**强制套古今双名；若日后景点也有对照表再扩展。
- **驿传「始末地」**：若关联京畿库藏，可显示双名；自由文本则保持用户输入（如「义乌」），不强制改写。

**未收录 / 自定义京畿**：

- 对照表无匹配时，允许 **御笔自建**：须手填 `ancientName` 与 `modernName`；展示仍用 `·` 连接。
- 若仅填一项：UI 降级为单项展示，保存时提示「宜补全古今双名，以合舆图体例」。

**实现备忘**：

```ts
// lib/grand-tour-labels.ts（示意）
export function formatTourRegionLabel(r: Pick<TourRegion, "ancientName" | "modernName">): string {
  const a = r.ancientName.trim();
  const m = r.modernName.trim();
  if (a && m) return `${a} · ${m}`;
  return a || m || "未命名京畿";
}
```

**M1 验收**：京畿列表、新建行在选目的地、北京样例行在概要，均显示 **燕京 · 北京市**；与 `城市古今名称对照清单.md` 中北京行一致。其余以 **§7.8** 为准。
